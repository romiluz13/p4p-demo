Slide 6 The Level 3 Structure 
## Context Loading Sequence
Before ANY task:
1. Read context/product-context.md
2. Read context/personas.md
3. Search notes/ for relevant background
4. Read context/tone-and-voice.md
5. Check context/examples/

## Agent Roles
When writing a PRD: act as a senior PM who has shipped this type before.
When analyzing data: act as a PM-trained analyst. Translate numbers to decisions.

## Anti-Patterns
NEVER start output with "Certainly!" or "Great!"
NEVER use passive voice in spec writing.
NEVER include "boilerplate" user stories.
"The anti-patterns section alone is worth the whole setup. If you've ever gotten Claude output that starts with 'Certainly!' ‚Äî this fixes that."
Slide 7 Context Hygiene: 3 Rules
1
Load at start, not mid-session
CLAUDE.md handles this automatically. Don't paste context mid-conversation. Start a new session instead.
2
Scope sessions to tasks, not topics
Bad: one long session for "Q3 planning"
Good: one session per deliverable (PRD, competitive brief, sprint review)
3
Persist decisions, not reasoning
After each session: update CLAUDE.md with what was decided ‚Äî not the conversation trail. Decisions are dense. Conversations are bloated.
"Most PM sessions are one long conversation about five different things. That's the worst possible use of context budget. One task. One session. End it. Start fresh."

Slide 10 The PM Pipeline
Task: write a PRD for Feature X
Step 1 (Researcher)
  Pull last 30 days of support tickets tagged Feature X
  ‚Üí save to research/feature-x-pain.md

Step 2 (Analyst)
  Read research file. What's the user's actual JTBD?
  ‚Üí save to research/feature-x-jtbd.md

Step 3 (Spec Writer)
  Read JTBD + CLAUDE.md. Write PRD using our template.
  ‚Üí save to specs/feature-x-prd.md

Step 4 (Reviewer)
  Does the PRD actually solve the pain from the research?
Each step outputs a file. Files are your handoff protocol.
"The reviewer step catches the most common PRD failure: a spec that's technically correct but doesn't solve the actual pain."

Slide 12 Handing Context Between Sessions
The #1 orchestration mistake: sessions don't share memory. Files do.
/project/
  CLAUDE.md              ‚Üê persistent brain
  context/               ‚Üê stable product info
  notes/                 ‚Üê working memory
  research/              ‚Üê session outputs
  specs/                 ‚Üê deliverables
  decisions/             ‚Üê decision log
Every session: what does this session produce?
Write it to the right folder. Next session reads the file, not your memory.
"Your project folder is your second brain. The structure isn't optional ‚Äî it's the handoff protocol between sessions."

Slide 18 The Review Loop
The most underused PM technique: asking Claude to review its own output.
"Now read this as the skeptical VP of Product.
What are the 3 weakest points?
Where would a senior engineer push back?
What assumption are we making that might fail?"
This is a mini Review Arena.
Engineers do this with 5 parallel reviewers.
You can do it in one extra prompt.
"The cc-teams plugin does this with multiple agents simultaneously. You can approximate it with one agent and a perspective-shift prompt."
