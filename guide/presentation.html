01
CONTEXT OPTIMIZATION
SLIDE 4 Context Is Your Real Bottleneck 
    Context window: the fixed resource you manage.
    What eats it:
    → Long back-and-forth without clearing            
    → Repeating product context every prompt          
    → Pasting the same background each session        
    → Asking Claude to hold too much at once          
    What happens when it fills: quality degrades.
    Claude starts guessing. Early context disappears.
    Context management = session quality management.
"The context window isn't infinite. When you fill it without structure, the model deprioritizes what was loaded first. That includes your CLAUDE.md instructions." Transition: the solution is CLAUDE.md architecture, not longer prompts.
SLIDE 5 CLAUDE.md as an Orchestration Tool 
    Most CLAUDE.md files are context dumps.
    The best ones are orchestration documents.
    LEVEL 1 (what most PMs have):                     
      • Product vision
      • Personas
      • Current OKRs
    LEVEL 3 (what this session gives you):            
      • Context loading sequence (explicit order)
      • Agent roles defined in the file
      • Response format rules
      • Anti-patterns (what Claude should NOT do)
      • Trigger → behavior mappings
The Level 3 is concrete — next slide shows the actual structure. Most PMs at Level 1 don't know what they're missing.
SLIDE 6 The Level 3 CLAUDE.md Structure 
   ## Context Loading Sequence
   Before ANY task:
   1. Read context/product-context.md
   2. Read context/personas.md
   3. Search notes/ for relevant background
   4. Read context/tone-and-voice.md
   5. Check context/examples/ — source of truth
   ## Agent Roles
   When writing a PRD: act as a senior PM
    who has shipped this type of feature before.
   When analyzing data: act as a PM-trained
    data analyst who translates numbers to decisions.
   ## Anti-Patterns
   NEVER start output with "Certainly!" or "Great!"
   NEVER use passive voice in spec writing.
   NEVER include "boilerplate" user stories.
Source: pm-studio (zoekdestep) — a PM workspace designed specifically for this. "The anti-patterns section alone is worth the whole setup. If you've ever gotten Claude output that starts with 'Certainly!' — this fixes that."
SLIDE 7 Context Hygiene: The 3 Rules 
    Rule 1: Load at the start, not mid-session
    CLAUDE.md handles this automatically.
    Don't paste context mid-conversation.
    Start a new session instead.
    Rule 2: Scope sessions to tasks, not topics
    Bad: one long session for "Q3 planning"
    Good: one session per deliverable
          (PRD, competitive brief, sprint review)
    Rule 3: Persist decisions, not reasoning
    After each session: update CLAUDE.md with
    what was decided — not the conversation trail.
    Decisions are dense. Conversations are bloated.
"Most PM sessions are one long conversation about five different things. That's the worst possible use of context budget. One task. One session. End it. Start fresh."
02
AGENT ORCHESTRATION
SLIDE 8 Agents for PMs (What This Actually Means) 
    "Agent" in PM context = a specialized Claude
    session with a defined role, tools, and memory.
    Agent types PMs actually use:
    RESEARCHER      Scrapes, synthesizes, reports
    SPEC WRITER     Drafts grounded in context
    DATA ANALYST    Queries, interprets, recommends   
    STRATEGIST      Cross-references, prioritizes
    REVIEWER        Audits output against standards
    Orchestration = running these in the right
    order with the right context handed between them.
"You don't need to install anything new for this. You're already doing it — just ad hoc, without structure. We're making the structure explicit."
SLIDE 9 The 3 Orchestration Patterns 
    PATTERN 1: SEQUENTIAL PIPELINE                    
    Research → Synthesize → Write spec → Review       
    Each step outputs a file. Next step reads it.
    PATTERN 2: PARALLEL RESEARCH                      
    "Research competitors A, B, C simultaneously"
    Claude Code's Task tool spawns parallel agents.
    Results merge into a comparison.
    PATTERN 3: SPECIALIST + SYNTHESIZER               
    Researcher gets the data.
    Analyst interprets it.
    PM Strategist writes the recommendation.
    (3 agents, 3 roles, one compound output)
Walk through a real example of each. For Sequential: PRD pipeline (next slide) For Parallel: competitive research (audience knows this) For Specialist + Synthesizer: retention analysis → insight → roadmap input
SLIDE 10 Live Pattern: The PM Pipeline 
    Task: write a PRD for Feature X
    AD HOC:          "Write a PRD for Feature X"      
                     [generic output]
    ORCHESTRATED:                                      
    Step 1 (Researcher): Pull last 30 days of
             support tickets tagged Feature X
             → save to research/feature-x-pain.md    
    Step 2 (Analyst): Read research file.
             What's the user's actual job to be done?
             → save to research/feature-x-jtbd.md    
    Step 3 (Spec Writer): Read JTBD + CLAUDE.md.
             Write PRD using our template.
             → save to specs/feature-x-prd.md        
    Step 4 (Reviewer): Does the PRD actually solve
             the pain from the research file?
"Each step outputs a file. The next step reads it. This is the core pattern. Files are your handoff protocol." The reviewer step catches the most common PRD failure: a spec that's technically correct but doesn't solve the actual pain.
SLIDE 11 Parallel Research in Practice 
    The prompt:
    "Research Competitor A, B, and C in parallel.
     For each: changelog last 90 days,
     top G2 complaints, recent job postings.
     Save each to research/competitor-[name].md
     Then synthesize into research/competitive.md"
    What Claude Code does:
    → Spawns 3 parallel research tasks (Task tool)    
    → Each writes to its own file                     
    → Final synthesis reads all three                 
    What used to take 3 days: 1-2 hours.
    What used to require a contractor: a prompt.
This is where Bright Data MCP becomes load-bearing. Without it: Claude Code reads local files. With it: the web is your data source. Point to modules/04-mcp-ecosystem.md for full setup reference.
SLIDE 12 Handing Context Between Sessions 
    The #1 orchestration mistake:
    forgetting that sessions don't share memory.
    The fix: treat files as your shared memory.
    /project/
      CLAUDE.md              ← persistent brain
      context/               ← stable product info
      notes/                 ← working memory
      research/              ← session outputs
      specs/                 ← deliverables
      decisions/             ← decision log
    Every session: what does this session produce?
    Write it to the right folder.
    Next session: reads the file, not your memory.
"Your project folder is your second brain. The structure isn't optional — it's the handoff protocol between sessions." Walk through each folder's purpose.
SLIDE 13 The Harness: What Engineers Have Built for This 
    Engineers solved this problem first.
    They call it a "harness" or "orchestration."
    cc-p4p (my PM plugin):
    → 1 router that reads your intent                 
    → 6 specialist agents (plan, build, debug...)     
    → Quality gate (RouterContract) every step        
    → Memory files that persist across sessions       
    The PM equivalent:
    → CLAUDE.md = the router's brain                  
    → Agent roles = specialist agents                 
    → File structure = memory system                  
    → Review step = quality gate                      
    You don't need cc-p4p. You need the PRINCIPLES.
This is the bridge slide between engineering harnesses and PM practice. "I built cc-p4p for PM workflows. The architecture is: Router → intent detection → specialist → quality gate → memory." PMs can instantiate this in CLAUDE.md without installing anything. "CLAUDE.md is not the only way. It's the foundation. Skills are the expertise layer on top." Skills fire automatically when relevant — prioritization skill activates when you're scoring features, user-research skill activates when you're synthesizing interviews. You don't invoke them, they show up. The official PM plugin's `/write-spec` runs a 5-step workflow: 5 questions → pulls from Jira + Notion + Figma → full PRD. That's not a template filler. That's a compound workflow triggered by one slash command.
03
MCP WORKFLOWS
SLIDE 14 MCPs Are Not Features. They're Workflow Nodes. 
    Most PM MCP usage: "I have Jira connected"
    Better PM MCP usage: Jira is a node in a flow
    ISOLATED MCP (what you probably do):
    → Open Claude Code                                
    → "Look at my Jira backlog"                       
    → Read the answer                                 
    MCP AS WORKFLOW NODE:                             
    → Sprint retro trigger                            
    → Read Jira sprint data                           
    → Read Amplitude: feature usage this sprint       
    → Read PRDs from Notion: what we said we'd build  
    → Compare all three                               
    → Output: planned vs built vs used analysis
"Every MCP is a data source. Orchestration = combining data sources into insights." This is the highest-leverage PM use of MCPs. Single MCP = data retrieval. Multiple MCPs = intelligence.
SLIDE 15 5 Compound MCP Workflows

       Jira (tickets) + Amplitude (usage) + Git

    2. COMPETITIVE INTELLIGENCE PIPELINE

ADVANCED PATTERNS
SLIDE 18 The Review Loop: Don't Ship Bad Output 
    The most underused PM technique:
    asking Claude to review its own output.
    After every major output, add:
    "Now read this as the skeptical VP of Product.
     What are the 3 weakest points?
     Where would a senior engineer push back?
     What assumption are we making that might fail?"
    This is a mini Review Arena.
    Engineers do this with 5 parallel reviewers.
    You can do it in one extra prompt.
"The cc-teams plugin does this with multiple agents simultaneously. You can approximate it with one agent and a perspective-shift prompt. Same effect, simpler setup."
SLIDE 19 PM Agent Roles Worth Defining 
    Define these in your CLAUDE.md:
    /research  → "Act as a rigorous researcher.       
                 Cite sources. Flag uncertainty."
    /spec      → "Act as a senior PM who's shipped    
                 this feature type before. Be
                 specific. Use our template."
    /data      → "Act as a PM-trained data analyst.   
                 Translate numbers to decisions.
                 Never bury the lede."
    /devil     → "Act as the skeptical VP who will    
                 question every assumption in this."
    These are slash commands you define.
    No plugin required.
"You put these in CLAUDE.md as instructions. When you type /spec, Claude knows what mode to enter. It's a custom command system built from plain text."
